Cuando ya tenemos muchos requerimientos, necesitamos organizarlos y decidir cuáles son más importantes. Es como cuando tienes una lista larga de compras pero poco dinero: ¿qué compras primero?

## ¿Por qué necesitamos analizar y priorizar?

<Aside type="caution">
**Realidad del desarrollo**: Casi nunca hay suficiente tiempo y dinero para hacer todo lo que el cliente quiere. Debemos elegir sabiamente qué hacer primero.
</Aside>

## Pasos para analizar requerimientos

Antes de priorizar, necesitamos entender bien cada requerimiento:

<Steps>
1. **Revisar claridad**: ¿Se entiende qué pide exactamente?
2. **Verificar completitud**: ¿Tiene toda la información necesaria?
3. **Comprobar factibilidad**: ¿Es posible hacerlo con la tecnología actual?
4. **Identificar dependencias**: ¿Depende de otros requerimientos?
5. **Estimar esfuerzo**: ¿Cuánto tiempo y recursos necesita?
</Steps>

### Ejemplo de análisis de un requerimiento:

<Tabs>
<TabItem label="Requerimiento Original">
"El sistema debe ser rápido"
</TabItem>
<TabItem label="Análisis">
- **Claridad**: ❗ Muy vago. ¿Qué significa "rápido"?
- **Completitud**: ❗ Falta información. ¿Qué partes deben ser rápidas?
- **Factibilidad**: ❓ No podemos evaluar sin más detalles
- **Dependencias**: ❓ Posiblemente del hardware disponible
- **Esfuerzo**: ❓ Imposible estimar sin aclaraciones
</TabItem>
<TabItem label="Requerimiento Mejorado">
"Las búsquedas de productos deben mostrar resultados en menos de 2 segundos, incluso con 1000 usuarios simultáneos"

- **Claridad**: ✅ Específico y medible
- **Completitud**: ✅ Indica función y criterio de éxito
- **Factibilidad**: ✅ Posible con optimización de base de datos
- **Dependencias**: ✅ Depende del diseño de la base de datos
- **Esfuerzo**: ✅ Estimado: 3 días de trabajo
</TabItem>
</Tabs>

### 1. Método MoSCoW

Es como clasificar tu lista de compras en: imprescindibles, importantes, deseables y descartados.

<Tabs>
<TabItem label="M - Must Have (Obligatorio)">
**Significado**: Requisitos fundamentales sin los cuales el sistema no funcionaría.

**Ejemplos**:
- Login de usuarios
- Proceso de pago
- Búsqueda de productos
- Carrito de compras

**Características**:
- Sin estos, el proyecto fracasa
- No son negociables
- Se implementan primero
</TabItem>
<TabItem label="S - Should Have (Debería)">
**Significado**: Importantes pero no vitales. El sistema puede funcionar sin ellos temporalmente.

**Ejemplos**:
- Filtros avanzados de búsqueda
- Historial de pedidos
- Recomendaciones personalizadas
- Notificaciones por email

**Características**:
- Agregan valor significativo
- Pueden esperar a una segunda fase
- Dolorosos de omitir, pero no fatales
</TabItem>
<TabItem label="C - Could Have (Podría)">
**Significado**: Deseables pero no necesarios. Son "extras" que mejoran la experiencia.

**Ejemplos**:
- Modo oscuro
- Compartir en redes sociales
- Animaciones en la interfaz
- Múltiples opciones de pago

**Características**:
- Menor impacto si se omiten
- Se implementan si sobra tiempo
- Mejoran la satisfacción pero no son esenciales
</TabItem>
<TabItem label="W - Won't Have (No tendrá)">
**Significado**: Descartados para esta versión, pero pueden considerarse en el futuro.

**Ejemplos**:
- Integración con asistentes de voz
- Realidad aumentada
- Programa de lealtad con puntos
- App para smartwatch

**Características**:
- Acordados como fuera de alcance
- Documentados para futuras versiones
- No se gastan recursos en ellos ahora
</TabItem>
</Tabs>

#### ¿Cómo usar la matriz?

<Steps>
1. **Estima el valor** de cada requerimiento (1-10)
2. **Estima el esfuerzo** necesario (1-10)
3. **Coloca cada requerimiento** en su cuadrante
4. **Prioriza** según la posición:
   - Cuadrante 1 (verde): Hazlo primero - son las "victorias rápidas"
   - Cuadrante 2 (amarillo): Planifica bien - valen la pena pero requieren esfuerzo
   - Cuadrante 3 (azul): Hazlo si hay tiempo - fáciles pero poco importantes
   - Cuadrante 4 (rosa): Evita o pospone - mucho trabajo para poco beneficio
</Steps>

<Aside type="tip">
**Consejo clave**: La validación no es solo preguntar "está bien", sino mostrar consecuencias reales de las decisiones de priorización.

### Técnicas efectivas para validar con el cliente

<Tabs>
<TabItem label="1. Demostración Visual">
- Usa prototipos para mostrar qué estará y qué no estará
- Presenta mockups de las funciones priorizadas
- Compara versiones "con" y "sin" ciertas funciones
- Muestra ejemplos de otros productos similares
</TabItem>
<TabItem label="2. Escenarios de Usuario">
- Cuenta historias de cómo usarán el producto
- "Con esta priorización, un usuario podrá hacer X pero no Y"
- Presenta un "día en la vida" usando el producto
- Muestra cómo se resolverán problemas reales
</TabItem>
<TabItem label="3. Datos y Evidencias">
- Presenta investigaciones de mercado
- Muestra estadísticas de uso de funciones similares
- Comparte feedback de pruebas con usuarios
- Calcula el retorno de inversión de cada función
</TabItem>
</Tabs>

#### Consejos para manejar desacuerdos:

<Steps>
1. **Escucha activamente**: Entiende por qué el cliente valora ciertos requerimientos
2. **Explica con ejemplos**: Usa casos reales de proyectos anteriores
3. **Ofrece alternativas**: "Si añadimos esto, podemos quitar aquello"
4. **Muestra el impacto**: "Añadir esta función retrasaría la entrega 2 semanas"
5. **Documenta las decisiones**: Anota los acuerdos y razones
</Steps>

## Resumen: El arte de priorizar

<Aside type="tip">
**Recuerda**: Priorizar no es decir "no" permanentemente, sino "ahora no". Es decidir el orden correcto para crear valor lo antes posible.
</Aside>

**Las claves para una buena priorización**:

1. **Analiza cada requerimiento** para entenderlo completamente
2. **Clasifica por importancia** usando métodos como MoSCoW
3. **Evalúa valor vs. esfuerzo** para maximizar el retorno
4. **Valida con el cliente** para asegurar alineación
5. **Documenta las decisiones** para referencia futura

Con estas herramientas, podrás tomar decisiones inteligentes sobre qué construir primero y entregar el máximo valor con los recursos disponibles.
