La notación UML (Unified Modeling Language) para clases proporciona un conjunto estándar de símbolos y convenciones que permiten representar clases y sus relaciones de manera clara y consistente. Esta notación es esencial para crear diagramas de clases efectivos que comuniquen correctamente la estructura de un sistema orientado a objetos.

## 4.1. Representación gráfica de una clase

En UML, una clase se representa mediante un rectángulo dividido en tres compartimentos o secciones. Cada sección tiene un propósito específico y sigue reglas de notación particulares.

### Estructura básica

<Code code={`[NombreClase]
-----------------
| atributos     |
-----------------
| operaciones   |
-----------------`} lang="text" title="Estructura básica de una clase en UML" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

class Persona {
  -nombre: String
  -edad: Integer
  +getNombre(): String
  +setNombre(nombre: String): void
}
@enduml`} />

### Compartimentos de una clase

<Tabs>
  <TabItem label="Compartimento superior">
    Contiene el **nombre de la clase** con las siguientes características:
    
    - Se escribe centrado
    - La primera letra en mayúscula (convención PascalCase)
    - Puede incluir el nombre del paquete: `paquete::NombreClase`
    - Se puede mostrar en **negrita** o *cursiva* para indicar propiedades especiales
      - *Cursiva*: Clase abstracta
      - **Negrita**: Clase estática
  </TabItem>
  
  <TabItem label="Compartimento intermedio">
    Contiene los **atributos** de la clase:
    
    - Cada atributo en una línea separada
    - Incluye visibilidad, nombre, tipo y valores iniciales
    - Puede omitirse si no es relevante para el diagrama
    - Puede incluir restricciones y propiedades adicionales
  </TabItem>
  
  <TabItem label="Compartimento inferior">
    Contiene las **operaciones** (métodos) de la clase:
    
    - Cada operación en una línea separada
    - Incluye visibilidad, nombre, parámetros y tipo de retorno
    - Puede omitirse si no es relevante para el diagrama
    - Puede incluir restricciones y propiedades adicionales
  </TabItem>
</Tabs>

### Variaciones en la representación

<Code code={`[NombreClase]
-----------------`} lang="text" title="Clase simplificada (solo nombre)" />

<Code code={`[NombreClase]
-----------------
| atributos     |
-----------------`} lang="text" title="Clase con nombre y atributos" />

<Code code={`[<<interfaz>>]
[NombreInterfaz]
-----------------
| operaciones   |
-----------------`} lang="text" title="Interfaz en UML" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

' Clase simplificada (solo nombre)
class Usuario

' Clase con nombre y atributos
class Producto {
  -id: Integer
  -nombre: String
  -precio: Double
}

' Interfaz en UML
interface Pagable {
  +pagar(): Boolean
  +calcularTotal(): Double
}

' Clase abstracta
abstract class Figura {
  #x: Integer
  #y: Integer
  +mover(dx: Integer, dy: Integer): void
  +{abstract} calcularArea(): Double
}

' Clase con estereotipo
class "<<entity>>\nCliente" as Cliente {
  -id: String
  -nombre: String
  -email: String
}
@enduml`} />

<Aside type="tip">
  En diagramas complejos, es común simplificar la representación de clases mostrando solo la información relevante para el contexto específico. Por ejemplo, en un diagrama que muestra principalmente relaciones, podría omitirse el compartimento de atributos y operaciones.
</Aside>

## 4.2. Representación de atributos

Los atributos en UML siguen una sintaxis específica que permite expresar diversas propiedades y características.

### Sintaxis completa de un atributo

<Code code={`visibilidad nombre: tipo [multiplicidad] = valorPredeterminado {propiedades}`} lang="text" title="Sintaxis de atributos en UML" />

### Elementos de un atributo

<Tabs>
  <TabItem label="Visibilidad">
    <div>
      <p>Indica el nivel de acceso al atributo:</p>
      <ul>
        <li><Badge text="+ Público" variant="success" /> - Accesible desde cualquier clase</li>
        <li><Badge text="- Privado" variant="danger" /> - Accesible solo desde la propia clase</li>
        <li><Badge text="# Protegido" variant="default" /> - Accesible desde la clase y sus subclases</li>
        <li><Badge text="~ Paquete" variant="note" /> - Accesible desde clases del mismo paquete</li>
      </ul>
    </div>
  </TabItem>
  
  <TabItem label="Nombre y tipo">
    <div>
      <ul>
        <li><strong>nombre</strong>: Identificador del atributo (generalmente en camelCase)</li>
        <li><strong>tipo</strong>: Tipo de dato del atributo (String, Integer, Boolean, etc.)</li>
        <li><strong>multiplicidad</strong>: Indica cuántas instancias puede tener (ej. [0..1], [1..*])</li>
      </ul>
    </div>
  </TabItem>
  
  <TabItem label="Valores y propiedades">
    <div>
      <ul>
        <li><strong>valorPredeterminado</strong>: Valor inicial asignado al atributo</li>
        <li>
          <strong>propiedades</strong>: Características adicionales entre llaves
          <ul>
            <li>{'{readOnly}'} - Atributo de solo lectura</li>
            <li>{'{unique}'} - Valor único</li>
            <li>{'{ordered}'} - Colección ordenada</li>
          </ul>
        </li>
      </ul>
    </div>
  </TabItem>
</Tabs>

### Ejemplos de atributos

<Code code={`- nombre: String
+ edad: Integer = 0
# saldo: Double {readOnly}
- cuentas: Cuenta[0..*] {ordered}
~ activo: Boolean = true`} lang="text" title="Ejemplos de atributos en UML" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

class Cliente {
  -nombre: String
  +edad: Integer = 0
  #saldo: Double {readOnly}
  -cuentas: Cuenta[0..*] {ordered}
  ~activo: Boolean = true
}

class Cuenta {
  -numero: String
  -tipo: String
  -fechaApertura: Date
}

Cliente "1" -- "0..*" Cuenta : tiene >

note right of Cliente::nombre
  Atributo privado
end note

note right of Cliente::edad
  Atributo público con valor inicial
end note

note right of Cliente::saldo
  Atributo protegido de solo lectura
end note

note right of Cliente::cuentas
  Colección ordenada de cuentas
end note
@enduml`} />

<Aside type="note">
  No es necesario incluir todos los elementos en cada atributo. La notación puede simplificarse según las necesidades del diagrama, mostrando solo la información relevante.
</Aside>

## 4.3. Representación de operaciones

Las operaciones (métodos) en UML siguen una sintaxis específica que permite expresar parámetros, tipos de retorno y otras propiedades.

### Sintaxis completa de una operación

<Code code={`visibilidad nombre(parametro1: tipo1, ..., parametroN: tipoN): tipoRetorno {propiedades}`} lang="text" title="Sintaxis de operaciones en UML" />

### Elementos de una operación

<Tabs>
  <TabItem label="Visibilidad y nombre">
    <div>
      <ul>
        <li><strong>visibilidad</strong>: Nivel de acceso (igual que en atributos)</li>
        <li><strong>nombre</strong>: Identificador del método (generalmente en camelCase)</li>
      </ul>
    </div>
  </TabItem>
  
  <TabItem label="Parámetros">
    <div>
      <p>Definidos entre paréntesis:</p>
      <ul>
        <li>
          <strong>dirección</strong>: Opcional, indica cómo se pasa el parámetro
          <ul>
            <li><code>in</code>: Parámetro de entrada (predeterminado)</li>
            <li><code>out</code>: Parámetro de salida</li>
            <li><code>inout</code>: Parámetro de entrada y salida</li>
          </ul>
        </li>
        <li><strong>nombre</strong>: Identificador del parámetro</li>
        <li><strong>tipo</strong>: Tipo de dato del parámetro</li>
        <li><strong>valorPredeterminado</strong>: Valor inicial opcional</li>
      </ul>
    </div>
  </TabItem>
  
  <TabItem label="Tipo de retorno y propiedades">
    <div>
      <ul>
        <li><strong>tipoRetorno</strong>: Tipo de dato que devuelve la operación</li>
        <li>
          <strong>propiedades</strong>: Características adicionales entre llaves
          <ul>
            <li>{'{query}'} - No modifica el estado del objeto</li>
            <li>{'{abstract}'} - Método abstracto (debe implementarse en subclases)</li>
            <li>{'{static}'} - Método estático (de clase)</li>
          </ul>
        </li>
      </ul>
    </div>
  </TabItem>
</Tabs>

### Ejemplos de operaciones

<Code code={`+ calcularTotal(): Double
- validarDatos(usuario: String, clave: String): Boolean
# actualizarSaldo(monto: Double): void
+ obtenerHistorial(): Transaccion[*] {query}
+ crearInstancia(): Producto {static}`} lang="text" title="Ejemplos de operaciones en UML" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

class CarritoCompra {
  -items: List<Item>
  -cliente: Cliente
  +calcularTotal(): Double
  -validarDatos(usuario: String, clave: String): Boolean
  #actualizarSaldo(monto: Double): void
  +obtenerHistorial(): Transaccion[*] {query}
}

class Producto {
  -codigo: String
  -nombre: String
  -precio: Double
  +{static} crearInstancia(): Producto
  +{abstract} calcularImpuestos(): Double
}

class Item {
  -cantidad: Integer
  -producto: Producto
  +getSubtotal(): Double
}

note right of CarritoCompra::calcularTotal
  Método público que devuelve un Double
end note

note right of CarritoCompra::validarDatos
  Método privado con parámetros
end note

note right of Producto::crearInstancia
  Método estático (de clase)
end note

note right of Producto::calcularImpuestos
  Método abstracto
end note
@enduml`} />

### Representación de constructores y destructores

<Code code={`+ NombreClase(parametros): void  // Constructor
+ ~NombreClase(): void         // Destructor`} lang="text" title="Constructores y destructores en UML" />

<Aside type="caution">
  En UML, los constructores tienen el mismo nombre que la clase, y los destructores se indican con un tilde (~) antes del nombre de la clase.
</Aside>

## 4.4. Estereotipos y etiquetas

Los estereotipos y etiquetas permiten extender la notación UML para representar conceptos específicos o propiedades adicionales.

### Estereotipos

Los estereotipos extienden el vocabulario de UML y se representan entre comillas angulares dobles (`<<estereotipo>>`).

<Code code={`[<<interfaz>>]
[InterfazUsuario]
-----------------
+ autenticar(): Boolean
+ cerrarSesion(): void
-----------------`} lang="text" title="Clase con estereotipo de interfaz" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

' Interfaz
interface "<<interface>>\nAutenticable" as Autenticable {
  +autenticar(usuario: String, clave: String): Boolean
  +cerrarSesion(): void
}

' Clase abstracta
abstract "<<abstract>>\nUsuario" as Usuario {
  #id: String
  #nombre: String
  +getNombre(): String
  +{abstract} obtenerPermisos(): List<String>
}

' Entidad
class "<<entity>>\nCliente" as Cliente {
  -email: String
  -telefono: String
}

' Controlador
class "<<control>>\nAutenticacionController" as AutenticacionController {
  +login(usuario: String, clave: String): Boolean
  +logout(): void
}

' Boundary
class "<<boundary>>\nLoginForm" as LoginForm {
  +mostrarFormulario(): void
  +procesarLogin(): void
}

' Utility
class "<<utility>>\nStringUtils" as StringUtils {
  +{static} formatear(texto: String): String
  +{static} validarEmail(email: String): Boolean
}

Usuario <|-- Cliente
Autenticable <|.. Cliente
LoginForm ..> AutenticacionController
AutenticacionController ..> Cliente
AutenticacionController ..> StringUtils
@enduml`} />

### Estereotipos comunes

| Estereotipo | Descripción | Aplicación |
|------------|-------------|------------|
| `&lt;&lt;interface&gt;&gt;` | Define una interfaz | Clase que solo declara métodos sin implementación |
| `&lt;&lt;abstract&gt;&gt;` | Clase abstracta | Clase que no puede ser instanciada directamente |
| `&lt;&lt;entity&gt;&gt;` | Entidad persistente | Clase que representa datos persistentes |
| `&lt;&lt;control&gt;&gt;` | Clase controladora | Clase que implementa lógica de negocio o control |
| `&lt;&lt;boundary&gt;&gt;` | Clase límite | Clase que interactúa con actores externos |
| `&lt;&lt;utility&gt;&gt;` | Clase utilitaria | Clase con métodos estáticos y sin estado |

### Etiquetas (Tagged Values)

Las etiquetas permiten añadir propiedades personalizadas a los elementos UML.

<Code code={`[Producto]
-----------------
| - id: String {id}
| - precio: Double
-----------------
{version = 1.0}
{author = "Juan Pérez"}`} lang="text" title="Clase con etiquetas" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

class Producto {
  -id: String {id}
  -precio: Double
}
note bottom of Producto
  {version = 1.0}
  {author = "Juan Pérez"}
end note

class Documento {
  -titulo: String
  -contenido: String
  -fechaCreacion: Date
}
note bottom of Documento
  {persistence = true}
  {table = "documentos"}
  {generated = true}
end note
@enduml`} />

### Restricciones

Las restricciones especifican condiciones que deben cumplirse y se representan entre llaves.

<Code code={`[Cuenta]
-----------------
| - saldo: Double {saldo >= 0}
-----------------
| + retirar(monto: Double): Boolean
-----------------`} lang="text" title="Clase con restricciones" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

class Cuenta {
  -saldo: Double {saldo >= 0}
  +retirar(monto: Double): Boolean {monto > 0}
  +depositar(monto: Double): void {monto > 0}
}

class Usuario {
  -nombre: String {minLength=3, maxLength=50}
  -email: String {unique}
  -edad: Integer {edad >= 18}
}

class Contraseña {
  -valor: String {minLength=8}
  +validar(): Boolean {!valor.contains(nombreUsuario)}
}

note right of Cuenta::saldo
  El saldo nunca puede ser negativo
end note

note right of Usuario::edad
  Solo se permiten usuarios mayores de edad
end note
@enduml`} />

<Aside type="tip">
  Los estereotipos, etiquetas y restricciones son mecanismos de extensión que permiten adaptar UML a dominios específicos o metodologías particulares. Son especialmente útiles cuando se trabaja con herramientas de generación de código o en contextos donde se requiere información adicional no cubierta por la notación estándar.
</Aside>

### Ejemplo completo de notación UML

<Code code={`[<<entity>>]
[Cliente]
-----------------
| + id: String {id, unique}
| - nombre: String [1]
| - apellido: String [1]
| - email: String [0..1]
| # fechaRegistro: Date = "hoy" {readOnly}
| ~ activo: Boolean = true
-----------------
| + getNombreCompleto(): String {query}
| + actualizarDatos(datos: DatosCliente): Boolean
| + desactivar(): void
| - validarEmail(email: String): Boolean
-----------------
{version = "2.0"}
{author = "Equipo de Desarrollo"}`} lang="text" title="Ejemplo completo de notación UML" />

<PlantUml code={`@startuml
skinparam classAttributeIconSize 0

package "com.empresa.clientes" {
  class "<<entity>>\nCliente" as Cliente {
    +id: String {id, unique}
    -nombre: String [1]
    -apellido: String [1]
    -email: String [0..1]
    #fechaRegistro: Date = "hoy" {readOnly}
    ~activo: Boolean = true
    +getNombreCompleto(): String {query}
    +actualizarDatos(datos: DatosCliente): Boolean
    +desactivar(): void
    -validarEmail(email: String): Boolean
  }
  
  class DatosCliente {
    -nombre: String
    -apellido: String
    -email: String
    -telefono: String
  }
  
  interface "<<interface>>\nEntidadPersistente" as EntidadPersistente {
    +getId(): String
    +esValido(): Boolean
  }
  
  abstract class "<<abstract>>\nPersona" as Persona {
    #nombre: String
    #apellido: String
    +getNombreCompleto(): String {query}
  }
  
  class Direccion {
    -calle: String
    -ciudad: String
    -codigoPostal: String
    -pais: String
  }
  
  ' Relaciones
  Cliente "1" o-- "0..*" Direccion : tiene >
  Cliente ..> DatosCliente : usa >
  EntidadPersistente <|.. Cliente
  Persona <|-- Cliente
}

note bottom of Cliente
  {version = "2.0"}
  {author = "Equipo de Desarrollo"}
end note

note right of Cliente::fechaRegistro
  Fecha en que se registró el cliente,
  no se puede modificar
end note
@enduml`} />

<Aside type="note">
  La notación UML es flexible y permite diferentes niveles de detalle según las necesidades del diagrama. En la práctica, es común omitir detalles que no son relevantes para el propósito específico del diagrama.
</Aside>
