Los diagramas de casos de uso se componen de tres elementos fundamentales: actores, casos de uso y las relaciones entre ellos. Estos elementos proporcionan una representación visual de cómo los usuarios interactúan con el sistema y qué funcionalidades ofrece este sistema.

## 2.1. Actores (usuarios y sistemas externos)

Los actores representan entidades externas que interactúan con el sistema. Pueden ser personas, organizaciones o incluso otros sistemas informáticos.

### Definición de actor

Un actor es cualquier entidad externa que intercambia información con el sistema. Los actores no forman parte del sistema en sí, sino que representan roles que interactúan con él.


### Tipos de actores

### Características de los actores

<div>
  <ul>
    <li><strong>Externos al sistema</strong>: Los actores siempre están fuera del límite del sistema.</li>
    <li><strong>Representan roles</strong>: Un actor es un rol, no una persona o entidad específica.</li>
    <li><strong>Pueden ser humanos o no humanos</strong>: Sistemas externos, dispositivos, sensores, etc.</li>
    <li><strong>Pueden tener relaciones entre sí</strong>: Principalmente relaciones de generalización/especialización.</li>
  </ul>
</div>

<Aside type="tip">
  Al identificar actores, pregúntese: ¿Quién o qué inicia acciones en el sistema? ¿Quién o qué proporciona o recibe información del sistema?
</Aside>

## 2.2. Casos de uso (funcionalidades del sistema)

Los casos de uso representan funcionalidades o servicios que el sistema proporciona a sus actores. Cada caso de uso describe una secuencia de acciones que el sistema realiza para producir un resultado observable y valioso para un actor.

### Definición de caso de uso

Un caso de uso es una descripción de una secuencia de interacciones entre el sistema y uno o más actores, que produce un resultado valioso y observable para los actores involucrados.


<PlantUml code={`@startuml
' Definición de casos de uso
usecase "Consultar Saldo" as ConsultarSaldo
usecase "Transferir Fondos" as TransferirFondos
usecase "Retirar Dinero" as RetirarDinero

' Relaciones entre casos de uso
RetirarDinero ..> ConsultarSaldo : <<include>>
RetirarDinero ..> TransferirFondos : <<extend>>

' Actor que interactúa con los casos de uso
actor Cliente
Cliente --> ConsultarSaldo
Cliente --> TransferirFondos
Cliente --> RetirarDinero
@enduml`} />

<Aside type="note">
  Este diagrama muestra cómo se representan los casos de uso en UML. Se muestran tres casos de uso (Consultar Saldo, Transferir Fondos y Retirar Dinero) con sus relaciones de inclusión y extensión, y un actor (Cliente) que interactúa con ellos.
</Aside>

### Características de los casos de uso

<div>
  <ul>
    <li><strong>Orientados a objetivos</strong>: Cada caso de uso representa un objetivo específico de un actor.</li>
    <li><strong>Iniciados por actores</strong>: Generalmente son desencadenados por una acción de un actor.</li>
    <li><strong>Proporcionan valor</strong>: Deben producir un resultado observable y valioso.</li>
    <li><strong>Completos</strong>: Describen una funcionalidad completa, desde el inicio hasta el final.</li>
  </ul>
</div>

  En los diagramas de casos de uso UML, solo se representa el nombre del caso de uso. Los detalles se documentan por separado en especificaciones de casos de uso.
</Aside>

### Límite del sistema

El límite del sistema es un rectángulo que engloba todos los casos de uso, representando la frontera entre el sistema y los actores externos.


<PlantUml code={`@startuml
rectangle "Sistema" {
  usecase "Caso de Uso 1" as CU1
  usecase "Caso de Uso 2" as CU2
  usecase "Caso de Uso 3" as CU3
}

actor Actor

Actor --> CU1
Actor --> CU2
Actor --> CU3
@enduml`} />

<Aside type="note">
  Este diagrama muestra el límite del sistema en UML. El rectángulo representa el sistema que contiene los casos de uso, mientras que los actores se encuentran fuera del sistema.
</Aside>

<div>
  <ul>
    <li>Define claramente qué está dentro y fuera del sistema.</li>
    <li>Ayuda a establecer el alcance del sistema.</li>
    <li>Todos los casos de uso deben estar dentro del límite.</li>
    <li>Todos los actores deben estar fuera del límite.</li>
  </ul>
</div>

## 2.3. Relaciones entre actores y casos de uso

Las relaciones en los diagramas de casos de uso conectan actores con casos de uso, o casos de uso entre sí, mostrando cómo interactúan.

### Asociación (actor-caso de uso)

La asociación es la relación más básica y común, que conecta un actor con un caso de uso indicando que el actor participa en ese caso de uso.


<PlantUml code={`@startuml
actor Usuario
usecase "Iniciar Sesión" as IniciarSesion

Usuario --> IniciarSesion
@enduml`} />

<Aside type="note">
  Este diagrama muestra la asociación entre un actor (Usuario) y un caso de uso (Iniciar Sesión). La flecha indica que el actor inicia la interacción con el caso de uso.
</Aside>

<div>
  <ul>
    <li>Se representa con una línea sólida entre el actor y el caso de uso.</li>
    <li>Indica que el actor se comunica con el sistema mediante este caso de uso.</li>
    <li>No tiene dirección específica (aunque a veces se usa una flecha para indicar quién inicia la interacción).</li>
    <li>Un actor puede asociarse con múltiples casos de uso.</li>
    <li>Un caso de uso puede asociarse con múltiples actores.</li>
  </ul>
</div>

<Aside type="tip">
  La asociación responde a la pregunta: ¿Qué actor interactúa con qué funcionalidad del sistema?
</Aside>

## Buenas prácticas al definir elementos básicos

<div>
  <ul>
    <li>
      <strong>Nombrar correctamente</strong>:
      <ul>
        <li>Actores: sustantivos que describan roles (Cliente, Administrador)</li>
        <li>Casos de uso: verbos en infinitivo + objeto (Registrar Usuario, Procesar Pago)</li>
      </ul>
    </li>
    <li><strong>Nivel adecuado de detalle</strong>: Evitar casos de uso demasiado generales o demasiado específicos</li>
    <li><strong>Enfoque en valor</strong>: Cada caso de uso debe proporcionar un resultado valioso para al menos un actor</li>
    <li><strong>Claridad en los límites</strong>: Definir claramente qué está dentro y fuera del sistema</li>
    <li><strong>Evitar diagramas sobrecargados</strong>: Si hay muchos casos de uso, considerar dividirlos en varios diagramas</li>
  </ul>
</div>

<Aside type="caution">
  Un error común es confundir pasos de un caso de uso con casos de uso independientes. Por ejemplo, "Introducir PIN" no debería ser un caso de uso separado, sino un paso dentro del caso de uso "Retirar Dinero".
</Aside>
