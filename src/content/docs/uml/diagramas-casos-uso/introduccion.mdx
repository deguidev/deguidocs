# Introducción al Diagrama de Casos de Uso

## Un ejemplo muy sencillo

Imaginemos que estamos diseñando una aplicación para un cajero automático. ¿Qué puede hacer un cliente con este cajero? Un diagrama de casos de uso nos ayuda a visualizarlo de manera simple:

<PlantUml code={`@startuml

actor "Cliente" as Cliente

rectangle "Cajero Automático" {
  usecase "Consultar Saldo" as ConsultarSaldo
  usecase "Retirar Dinero" as RetirarDinero
  usecase "Depositar Dinero" as DepositarDinero
  usecase "Cambiar PIN" as CambiarPIN
}

Cliente --> ConsultarSaldo
Cliente --> RetirarDinero
Cliente --> DepositarDinero
Cliente --> CambiarPIN

@enduml`} />

En este sencillo ejemplo:

- Tenemos un **actor**: el `Cliente` que usa el cajero
- Tenemos cuatro **casos de uso** (funcionalidades que ofrece el sistema):
  - Consultar saldo
  - Retirar dinero
  - Depositar dinero
  - Cambiar PIN
- Las **líneas** conectan al actor con los casos de uso que puede realizar

Este diagrama nos permite entender rápidamente:
- Qué puede hacer un usuario con el sistema
- Cuáles son las funcionalidades principales que debemos implementar
- Los límites del sistema (lo que está dentro del rectángulo)

Sin necesidad de código o detalles técnicos, cualquier persona puede comprender qué hará nuestro sistema.

## Definición y propósito

Los diagramas de casos de uso son una herramienta fundamental en el modelado de sistemas y la ingeniería de requisitos. Representan gráficamente la interacción entre los usuarios (actores) y el sistema, mostrando las funcionalidades que el sistema debe proporcionar y cómo los diferentes tipos de usuarios interactúan con estas funcionalidades.

## 1.1. Definición y propósito

### ¿Qué es un diagrama de casos de uso?

Un diagrama de casos de uso es una representación gráfica de las interacciones entre los usuarios (actores) y el sistema. Forma parte del lenguaje UML (Unified Modeling Language) y se centra en describir cómo se utiliza el sistema desde la perspectiva del usuario.

<Code code={`+---------------------+
|                     |
|      Sistema        |
|                     |
| +---------------+   |
| |               |   |
| | Caso de Uso 1 |<--+--- Actor 1
| |               |   |
| +---------------+   |
|                     |
| +---------------+   |
| |               |   |
| | Caso de Uso 2 |<--+--- Actor 2
| |               |   |
| +---------------+   |
|                     |
+---------------------+`} lang="text" title="Estructura básica de un diagrama de casos de uso" />

<PlantUml code={`@startuml

rectangle "Sistema de Biblioteca" {
  usecase "Prestar Libro" as UC1
  usecase "Devolver Libro" as UC2
  usecase "Buscar Libro" as UC3
  usecase "Gestionar Multas" as UC4
}

actor Bibliotecario
actor "Usuario" as User
actor "Administrador" as Admin

Bibliotecario --> UC1
Bibliotecario --> UC2
User --> UC3
Bibliotecario --> UC4
Admin --> UC4

@enduml`} />

### Propósito principal

<div>
  <ul>
    <li><strong>Capturar requisitos funcionales</strong>: Identificar qué debe hacer el sistema desde la perspectiva del usuario.</li>
    <li><strong>Definir el alcance del sistema</strong>: Establecer claramente los límites del sistema y lo que queda fuera de él.</li>
    <li><strong>Facilitar la comunicación</strong>: Proporcionar un lenguaje común entre usuarios, clientes y desarrolladores.</li>
    <li><strong>Guiar el desarrollo</strong>: Servir como base para la planificación y el diseño del sistema.</li>
  </ul>
</div>

<Aside type="tip">
  Los diagramas de casos de uso son especialmente útiles en las etapas iniciales del desarrollo, cuando es crucial entender qué necesita hacer el sistema antes de decidir cómo lo hará.
</Aside>

### Enfoque centrado en el usuario

<div>
  <ul>
    <li>Priorizan las necesidades y expectativas de los usuarios finales.</li>
    <li>Permiten identificar diferentes tipos de usuarios y sus interacciones específicas con el sistema.</li>
    <li>Ayudan a validar que el sistema satisfará las necesidades reales de los usuarios.</li>
  </ul>
</div>

### Clarificación de requisitos

<div>
  <ul>
    <li>Transforman requisitos textuales en representaciones visuales más fáciles de entender.</li>
    <li>Facilitan la identificación de requisitos incompletos, ambiguos o contradictorios.</li>
    <li>Permiten descubrir nuevos requisitos durante el proceso de modelado.</li>
  </ul>
</div>

</Aside>

## 1.3. Relación con otros diagramas UML

Los diagramas de casos de uso no existen de forma aislada, sino que se complementan con otros diagramas UML para proporcionar una visión completa del sistema.

<PlantUml code={`@startuml

package "Diagramas UML" {
  package "Diagramas de Comportamiento" {
    rectangle "Diagrama de Casos de Uso" as DCU #LightBlue
    rectangle "Diagrama de Actividades" as DA
    rectangle "Diagrama de Secuencia" as DS
    rectangle "Diagrama de Estados" as DE
  }
  
  package "Diagramas Estructurales" {
    rectangle "Diagrama de Clases" as DC
    rectangle "Diagrama de Componentes" as DCO
    rectangle "Diagrama de Despliegue" as DD
  }
}

DCU -down-> DA : "detalla flujo de\ntrabajos de casos de uso"
DCU -down-> DS : "muestra interacción\nentre objetos"
DCU -down-> DE : "representa estados\nde objetos"

DCU -down-> DC : "define clases para\nimplementar casos de uso"
DCU -down-> DCO : "organiza componentes\nque implementan casos de uso"
DCU -down-> DD : "especifica infraestructura\nfísica"

@enduml`} />

### Flujo de trabajo típico en UML

<Code code={`Requisitos → Casos de Uso → Diagramas de Comportamiento → Diagramas Estructurales → Implementación`} lang="text" title="Flujo de trabajo típico en el modelado UML" />

<Aside type="tip">
  Aunque este flujo representa una progresión lógica, en la práctica el desarrollo suele ser iterativo, con refinamientos continuos en todos los diagramas a medida que evoluciona el entendimiento del sistema.
</Aside>

## Beneficios clave de los diagramas de casos de uso

<div>
  <ul>
    <li><strong>Simplicidad</strong>: Son fáciles de entender incluso para personas sin conocimientos técnicos.</li>
    <li><strong>Visión global</strong>: Proporcionan una vista panorámica de las funcionalidades del sistema.</li>
    <li><strong>Priorización</strong>: Ayudan a identificar las funcionalidades más importantes para los usuarios.</li>
    <li><strong>Trazabilidad</strong>: Permiten rastrear requisitos desde su origen hasta su implementación.</li>
    <li><strong>Gestión del alcance</strong>: Facilitan la definición clara de lo que está dentro y fuera del alcance del proyecto.</li>
  </ul>
</div>
