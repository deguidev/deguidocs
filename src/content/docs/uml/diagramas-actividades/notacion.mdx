# Notación UML para Diagramas de Actividades

La notación UML para diagramas de actividades proporciona un conjunto estandarizado de símbolos y convenciones para representar flujos de trabajo y procesos. Esta guía se basa en la sintaxis beta de PlantUML que ofrece mayor flexibilidad y características avanzadas.

## 4.1. Elementos fundamentales

Los diagramas de actividades utilizan elementos específicos para representar diferentes aspectos del flujo de trabajo.

<PlantUml code={`@startuml
start
:Acción simple;
note right: Actividad básica

:Objeto de datos; <<object>>
note right: Nodo de objeto

:Enviar señal; <<sendSignal>>
note right: Acción de envío

:Esperar evento; <<acceptEvent>>
note right: Acción de recepción

:Timeout 30s; <<timeEvent>>
note right: Evento temporal

if (¿Condición?) then (verdadero)
  :Acción A;
else (falso)
  :Acción B;
endif

fork
  :Proceso paralelo 1;
fork again
  :Proceso paralelo 2;
end fork

stop
@enduml`} />

## 4.2. Sintaxis de actividades básicas

Las actividades son los elementos fundamentales y se definen con una sintaxis específica en PlantUML.

</Aside>

## 4.3. Estructuras de control

Las estructuras de control permiten representar decisiones, bucles y flujos condicionales en los diagramas de actividades.

### 4.3.1. Decisiones condicionales

<PlantUml code={`@startuml
start
:Recibir solicitud;

if (¿Usuario autenticado?) then (sí)
  :Procesar solicitud;
  
  if (¿Datos válidos?) then (sí)
    :Guardar en base de datos;
    :Enviar confirmación;
  else (no)
    :Mostrar errores de validación;
  endif
  
else (no)
  :Solicitar autenticación;
  :Redirigir a login;
endif

stop
@enduml`} />

### 4.3.2. Bucles y repeticiones

<PlantUml code={`@startuml
start
:Inicializar contador = 0;

while (¿contador < 10?) is (sí)
  :Procesar elemento;
  :Incrementar contador;
endwhile (no)

:Finalizar proceso;

repeat
  :Solicitar entrada del usuario;
  :Validar entrada;
repeat while (¿Entrada inválida?) is (sí)

stop
@enduml`} />

### Sintaxis de estructuras de control

<Tabs>
  <TabItem label="Decisiones">
    <Code code={`if (condición) then (verdadero)
  :acción si verdadero;
else (falso)
  :acción si falso;
endif

' Decisión múltiple
if (condición) then (opción1)
  :acción para opción1;
else if (opción2)
  :acción para opción2;
else (otra)
  :acción por defecto;
endif`} lang="text" title="Sintaxis de decisiones" />
  </TabItem>
  <TabItem label="Bucle while">
    <Code code={`while (condición) is (verdadero)
  :actividad repetitiva;
endwhile (falso)`} lang="text" title="Sintaxis de bucle while" />
  </TabItem>
  <TabItem label="Bucle repeat">
    <Code code={`repeat
  :actividad repetitiva;
repeat while (condición) is (verdadero)`} lang="text" title="Sintaxis de bucle repeat" />
  </TabItem>
  <TabItem label="Bucle foreach">
    <Code code={`foreach item in collection
  :procesar item;
end`} lang="text" title="Sintaxis de bucle foreach" />
  </TabItem>
</Tabs>

<Aside type="note">
  Las etiquetas en los flujos de decisión deben ser claras y corresponder a los posibles resultados de la condición evaluada.
</Aside>

## 4.4. Flujos paralelos y sincronización

Los flujos paralelos permiten representar actividades que se ejecutan concurrentemente usando fork/join.

<PlantUml code={`@startuml
start
:Iniciar proceso;

fork
  :Validar datos;
  :Procesar información A;
fork again
  :Verificar permisos;
  :Procesar información B;
fork again
  :Registrar auditoría;
  :Generar log;
end fork

:Consolidar resultados;
:Finalizar proceso;
stop
@enduml`} />

### 4.4.1. Terminación condicional de flujos

<PlantUml code={`@startuml
start
:Iniciar descarga;
note right: Proceso principal de descarga

fork
  :Descargar archivo;
  note right: Hilo principal de descarga
  :Verificar integridad;
  note right: Validación del archivo
fork again
  :Mostrar progreso;
  note right: Hilo de interfaz de usuario
  while (¿Descarga completa?) is (no)
    :Actualizar barra de progreso;
  endwhile (sí)
fork again
  :Esperar cancelación del usuario;
  note right: Hilo de monitoreo de cancelación
  if (¿Usuario canceló?) then (sí)
    :Limpiar recursos;
    note right: Limpieza antes de terminar
    kill
  else (no)
    :Continuar;
  endif
end fork

:Notificar descarga completa;
note right: Consolidación de resultados
stop
@enduml`} />

### Sintaxis y variantes

<Tabs>
  <TabItem label="Fork/Join simple">

fork again
  :actividad rama 2;
end fork`} lang="text" title="Sintaxis básica de fork/join" />

## 4.8. Notación de bucles

Los bucles representan actividades que se repiten hasta que se cumple una condición específica.

<PlantUml code={`@startuml
start
:Inicializar contador = 0;

:Finalizar proceso;
note right: Después del bucle while

  :Solicitar entrada del usuario;
  note right: Actividad que se repite
  :Validar entrada;
  note right: Validación de datos
repeat while (¿Entrada inválida?) is (verdadero)

:Procesar entrada válida;
note right: Procesamiento final
stop
@enduml`} />

### Sintaxis y variantes

<Tabs>
  <TabItem label="While">

## 4.9. Notación de excepciones

La notación de excepciones representa el manejo de situaciones excepcionales o errores durante la ejecución del proceso.

<PlantUml code={`@startuml
start
:Iniciar operación;
note right: Preparación para operación riesgosa

if (¿Operación segura?) then (sí)
  :Ejecutar operación normal;
  note right: Flujo normal sin errores
else (no)
  :Detectar error;
  note right: Identificación del problema
  
  if (¿Error crítico?) then (sí)
    :Registrar error crítico;
    note right: Log de error crítico
    :Notificar administrador;
    note right: Escalamiento del problema
    stop
  else (no)
    :Registrar advertencia;
    note right: Log de advertencia
    :Continuar con precaución;
    note right: Flujo alternativo
  endif
endif

:Finalizar operación;
note right: Conclusión exitosa
stop
@enduml`} />

### Sintaxis y variantes

<Tabs>
  <TabItem label="Manejo con decisiones">

  :operación normal;
else (no)
  :manejar error;

  </TabItem>

  endif

## 4.12. Buenas prácticas de notación

Seguir buenas prácticas de notación es esencial para crear diagramas de actividades claros y efectivos.

### Recomendaciones generales

<div>
  <ul>
    <li><strong>Nombres descriptivos</strong>: Usa nombres concisos pero descriptivos para actividades y objetos.</li>
    <li><strong>Nivel de detalle adecuado</strong>: Incluye suficiente detalle para comprender el proceso, pero evita el exceso que puede complicar el diagrama.</li>
    <li><strong>Consistencia</strong>: Mantén un estilo consistente en todo el diagrama (nomenclatura, colores, etc.).</li>
    <li><strong>Dirección del flujo</strong>: Mantén un flujo general de arriba hacia abajo o de izquierda a derecha para facilitar la lectura.</li>
    <li><strong>Alineación</strong>: Alinea los elementos para mejorar la legibilidad y la estética del diagrama.</li>
    <li><strong>Uso de colores</strong>: Usa colores para destacar elementos importantes o diferenciar tipos de actividades, pero sin exceso.</li>
  </ul>
</div>

### Errores comunes a evitar

<div>
  <ul>
    <li><strong>Diagramas sobrecargados</strong>: Evita incluir demasiados elementos en un solo diagrama.</li>
    <li><strong>Flujos cruzados</strong>: Minimiza el cruce de líneas que dificultan la lectura del diagrama.</li>
    <li><strong>Inconsistencia en la terminología</strong>: Usa los mismos términos para referirte a los mismos conceptos.</li>
    <li><strong>Falta de contexto</strong>: Proporciona suficiente información para entender el propósito del diagrama.</li>
    <li><strong>Ambigüedad en las decisiones</strong>: Asegúrate de que las condiciones y resultados en los puntos de decisión sean claros.</li>
  </ul>
</div>

<PlantUml code={`@startuml
' Ejemplo de buenas prácticas
skinparam backgroundColor white
skinparam activity {
  BackgroundColor #lightBlue
  BorderColor #blue
}

|#lightgreen|Cliente|
start
:Realizar pedido;
|#lightblue|Sistema|
:Validar pedido;
if (¿Pedido válido?) then (sí)
  :Procesar pago;
  note right: Verificar fondos disponibles
  if (¿Pago exitoso?) then (sí)
    |#lightyellow|Almacén|
    :Preparar pedido;
    :Enviar pedido;
    |#lightgreen|Cliente|
    :Recibir pedido;
  else (no)
    |#lightgreen|Cliente|
    :Notificar problema de pago;
  endif
else (no)
  |#lightgreen|Cliente|
  :Notificar error en pedido;

@enduml`} />

<Aside type="note">
  Un buen diagrama de actividades debe ser comprensible por sí mismo, sin necesidad de explicaciones adicionales extensas. Debe comunicar claramente el flujo del proceso y las decisiones involucradas.
</Aside>

La notación UML para diagramas de actividades proporciona un lenguaje visual estandarizado que facilita la comunicación y documentación de procesos. Dominar esta notación te permitirá crear diagramas efectivos que sean fáciles de entender para todos los interesados en el proyecto.
